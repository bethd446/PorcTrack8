Notes design

3 décisions · 2 hésitations · effort dev

3 décisions design fortes

01 · Onboarding

Slide-X plein écran · pas stack scrollable

Wizard 5 étapes plein écran, animation slide horizontale entre étapes, progress 5 barres en haut. Stack scrollable aurait dilué l'engagement et caché l'avancement. Skip top-right discret, sans confirmation — la friction est l'inverse de ce qu'on veut au démarrage. Skip total et non step-by-step : pas de clics répétés.

02 · Modals

Sticky footer 2 boutons · pas FAB

Sticky footer Annuler ghost + Action primary. FAB écarté : il flotte au-dessus du contenu et masque la dernière ligne de form. Sticky footer = touch target garanti 48px, label complet ("Enregistrer la truie", pas juste un check), bouton Annuler symétrique. Pattern réutilisable pour tous les sheets.

03 · Datepicker · Autocomplete

Natif HTML5 + autocomplete custom 3 résultats

Datepicker = input type="date" natif Android Chrome, supérieur au custom V70 sur mobile (calendrier OS, locale auto). Autocomplete custom : 3 résultats max visibles, avatar coloré + code mono + sub métier (J113 · saillie 17/01) — bien plus utile qu'une liste alphabétique pour 50 truies.

2 zones d'hésitation

Hésité · Onboarding C.4

Saillie réelle ou simulation ?

Tenté : la saillie de C.4 écrit vraiment T-001 + cycle dans Supabase pour que Yao retrouve son cycle après onboarding. Risque : si Yao skip après C.4, la donnée orpheline pollue son élevage. Choisi : simulation pure, animation cycle qui se peuple visuellement, toast success — mais aucune écriture DB. C.5 redirige vers /today qui montre l'état réel (vide pour un nouveau compte).

Hésité · Modal Bande

Truie mère en autocomplete ou chip-list ?

Chip-list (3 truies J100+ visibles d'office) aurait économisé un tap. Mais 3 truies ≠ règle : à terme une ferme à 50 truies aura 6-8 J100+ simultanées. Autocomplete avec input pré-focusé + 3 suggestions filtrées dès l'ouverture = compatible 3 truies (zéro frappe) ET 8 truies (1-2 frappes). Pattern unique pour toutes les fermes.

Effort dev · React/Ionic Capacitor

Onboarding wizard

~4 j · 5 étapes

Component <OnboardingWizard /> avec slide animation, 5 routes ou 1 route + state. Le mock-form C.3/C.4 n'écrit pas en DB — pure UI. Inclut illustration silhouette truie SVG inline (réutilisable). Hors traduction baoulé/dioula.

Sheets quick-add

~5 j · 6 modals + autocomplete

1 j base sheet drag-handle (Ionic IonModal breakpoints=[0, 0.9]) + 4 j × form fields. <TruieAutocomplete /> et <BandeAutocomplete /> = composants partagés. Stepper numérique = atom déjà existant. Cycle preview de D.6 réutilise le pattern .cycle-prev direct.

Cumul 4 mockups

~41 j / 1 dev

14 j Élevage + 9 j Réglages-Pilot + 9 j Ressources-Repro + 9 j Onboarding-Modals. Hors backend Supabase (auth, RLS, triggers cycle), QA terrain Yamoussoukro, traduction. Top 3 priorités cette passe : (1) sheets quick-add — ce que Yao tape 20 fois/jour, (2) onboarding C.4 saillie — fondation du cycle vivant, (3) autocomplete truie — composant partagé qui débloque saillie + bande + MB.
